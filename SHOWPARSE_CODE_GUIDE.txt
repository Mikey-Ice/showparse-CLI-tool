1. READ THIS FIRST
===============================================================================

SHORT VERSION:
    showparse extracts prompt-delimited command output from network collection
    files. The parser is intentionally strict because false positives are worse
    than missing a vague match.

CURRENT STABLE CLI FLAGS:
    -q    standard query mode
    -Q    block-aware query mode
    -A    require all queries to return content for a file
    -r    raw grep-style output
    --no-color    print the per-file filename banner without ANSI color
    --no-banner    suppress the per-file filename banner in normal mode
    -n    notes mode with deduplication
    -o    save the final notes report to a file (notes mode only)

CURRENT MODIFIERS:
    %    matched substring only
    ~    show matched command first
    #    count non-empty rendered lines after all other modifiers
    +    all prompt-delimited command matches for that query
    @    parent + matched child lines only (-Q only)
    /    case-sensitive pattern matching

MOST IMPORTANT RULE IN THE WHOLE TOOL:
    Query commands start with a letter.
    Modifiers are symbolic non-letter characters only.

WHY THAT RULE EXISTS:
    It keeps modifier parsing simple and safe.
    If the first character is a letter, the query starts with the command.
    If the first character is a defined symbolic modifier, everything up to the
    first colon is treated as the modifier set.

FUTURE MODIFIER RULE:
    Do not add letter-based modifiers.
    Keep modifiers symbolic so command-vs-modifier detection stays unambiguous.


===============================================================================
2. WHAT SHOWPARSE IS FOR
===============================================================================

showparse is built for real-world network collection files where device output
has been captured as a sequence of command prompts and command output.

EXPECTED FILE SHAPE:
    Router1#show version
    <output lines>
    Router1#show running-config
    <output lines>

    switch1# show version
    <output lines>
    switch1# show running-config
    <output lines>

PRIMARY JOB:
    Extract one or more command outputs safely, then optionally filter them
    into the exact shape needed for field analysis or notes/reporting.

WHY THE TOOL EXISTS:
    Standard grep is too loose for these captures because commands like
    `show run` can appear inside `show history`, logs, or other command output.
    showparse first finds the real command invocation at the prompt, then
    filters inside the extracted output. That is the foundation of the tool.

PRODUCT GOAL:
    Make it easy to cut deeply into collection files while still producing
    stable, normalized output that groups well in notes mode.


===============================================================================
3. STABLE CLI SURFACE
===============================================================================

STANDARD QUERY MODE:
    -q "show version"
        Extract full output for one command

    -q "show run:hostname|username"
        Extract command output and return matching lines only

    -q "~:show run:username"
        Return the real matched command line first, then the rendered output

    -q "%:show logging:Trap logging: level \\w+"
        Extract command output and return only the matched substring from each
        matching line

    -q "#:show version"
        Extract command output normally, then replace it with the number of
        non-empty lines that would have been displayed

    -q "+:show run:hostname|username|logging host"
        Run the query against every prompt-delimited command that matches the
        command prefix

BLOCK QUERY MODE:
    -Q "show run:shutdown"
        Return full parent/child config blocks containing a match

    -Q "@:show run:switchport mode access|shutdown"
        Return parent + matched child lines only unless the parent itself
        matched

    -Q "~@:show run:switchport mode access|shutdown"
        Return the real matched command line first, then the selective block
        output

    -Q "%:show run:shutdown"
        Keep full block context but trim matching lines down to the matched
        substring only

OUTPUT FLAGS:
    -A
        AND mode. All queries must return content for a file to be shown.

    -r
        Raw mode. Prefix each non-empty output line with `filename:`.

    --no-color
        Normal mode only. Keep the per-file filename banner, but print it as
        plain text instead of yellow ANSI text.

    --no-banner
        Normal mode only. Suppress the yellow per-file filename banner.
        Multiple files are still separated by a blank line.

    -n
        Notes mode. Deduplicate identical rendered outputs, review each unique
        output once, and collect notes.

    -o PATH
        Notes mode only. Save the final NOTES REPORT block to a file instead of
        printing it to stdout. The report includes a compact `Command:` line
        showing reusable showparse syntax, not file targets or notes/save flags.

REMOVED V2 FLAGS:
    -c, -g, -G

CURRENT HELP/DOC MODEL:
    The CLI is intentionally query-centric now. All extraction and filtering
    flows through -q and -Q.


===============================================================================
4. QUERY GRAMMAR LAW
===============================================================================

BASIC SHAPES:
    -q "command"
    -q "command:pattern"
    -q "modifiers:command:pattern"

    -Q "command:pattern"
    -Q "modifiers:command:pattern"

NOTE:
    The code currently allows `-Q "command"` and `-Q "+:command"` because the
    same parser is used underneath. In practice, -Q is intended for block-aware
    queries and is normally used with a pattern.

COLON RULE:
    Only the first colon after the command split matters.
    Patterns may still contain colons.

EXAMPLES:
    "show version"
        command = show version
        pattern = None

    "show run:interface:Ethernet0/1"
        command = show run
        pattern = interface:Ethernet0/1

    "+%:show logging:Trap logging: level \\w+"
        modifiers = +%
        command = show logging
        pattern = Trap logging: level \\w+

MODIFIER COMPATIBILITY:
    -q supports:
        %  ~  #  +  /

    -Q supports:
        %  ~  #  +  @  /

VALIDATION RULES:
    - `@` is rejected on -q
    - `%` requires a pattern
    - `/` requires a pattern
    - `@` requires a pattern
    - `+` can be used with or without a pattern
    - `~` can be used with or without a pattern

ORDER RULE:
    Modifier order does not matter.
    `+%`, `%+`, `/%`, `%/`, `+@%/`, etc. are treated as sets of modifiers.


===============================================================================
5. CORE PARSING PHILOSOPHY
===============================================================================

THIS IS THE SINGLE MOST IMPORTANT DESIGN SECTION.

FOUNDATION:
    showparse is prompt-delimited first, grep-like second.

COMMAND START RULE:
    A command match is valid only when it appears at column 0 with a prompt
    token immediately before it, or with exactly one space after the prompt.

SUPPORTED PROMPT STYLES:
    Router1#show run
    Router1# show run

NOT SUPPORTED:
    Router1#  show run
    "  show run" inside indented output

WHY SO STRICT:
    The tool is deliberately designed to avoid matching indented `show history`
    entries, log text, or other command-looking strings that are not actual
    prompt-delimited command invocations.

PROMPT TOKEN RULE:
    Prompt detection uses a token ending in a common prompt character like
    `#`, `>`, `$`, or `%`. This keeps metadata lines such as `!Command:` from
    being treated as device prompts.

COMMAND MATCH RULE:
    Command extraction is case-insensitive and prefix-based.

EXAMPLE:
    Querying `show run` can match:
        show running-config
        show running-config all

OUTPUT BOUNDARY RULE:
    Once a command match is found, output continues until the same prompt token
    appears again at column 0 later in the file.

DO NOT CASUALLY CHANGE:
    - prompt must stay at column 0
    - same-prompt boundary behavior
    - zero-or-one-space support after the prompt
    - case-insensitive prefix-based command extraction

6. MODIFIER INTENT
===============================================================================

The modifiers are important because they let the user shape output into forms
that normalize well in notes mode.

%  MATCHED SUBSTRING ONLY
    Purpose:
        Strip matching lines down to just the matched regex text.

    Why it exists:
        Real outputs often have useful data followed by noisy counters or
        trailing text. `%` helps normalize those lines so similar devices group
        more cleanly under `-n`.

    Example:
        `Trap logging: level informational, 73 message lines logged`
        becomes:
        `Trap logging: level informational`

#  LINE COUNT
    Purpose:
        Replace successful rendered output with the number of non-empty lines
        that would have been displayed.

    Why it exists:
        Sometimes the user wants a quick volume check rather than the actual
        content. `#` provides a field-friendly `wc -l` style result while still
        honoring showparse's prompt-delimited extraction and filtering rules.

    Important:
        `#` runs after all other output shaping for that query.
        It does not count blank separator lines.
        It does not convert command-not-found messages into counts.

~  SHOW MATCHED COMMAND FIRST
    Purpose:
        Prepend the actual matched command line from the file before the
        rendered output for that command result.

    Why it exists:
        Prefix queries like `show run` can match different real commands such
        as `show running-config` and `show running-config all`. `~` makes the
        source command explicit in the output.

    Important:
        The injected line removes the device prompt but keeps the real matched
        command text.
        With `+`, one command line is injected for each matched command result.
        `#` counts the injected line because it counts final rendered output.

+  ALL MATCHING COMMANDS
    Purpose:
        Instead of taking the first prompt-delimited command match only, process
        every prompt-delimited command whose command text matches the query
        prefix.

    Why it exists:
        `show run` is often shorthand for both `show running-config` and
        `show running-config all`. `+` lets the user intentionally widen that.

@  PARENT + MATCHED CHILDREN ONLY
    Purpose:
        Narrow block-aware output so that child matches do not drag along every
        sibling line in the block.

    Why it exists:
        Plain `-Q` is great for full context, but sometimes it is too noisy.
        `@` gives an in-between mode:
            parent matched -> full block
            child matched -> parent + matching child lines only

    Scope:
        `@` is `-Q` only.

/  CASE-SENSITIVE PATTERN MATCHING
    Purpose:
        Make the grep/block pattern case-sensitive for that query only.

    Why it exists:
        Sometimes the only reliable way to separate similar lines is case.

    Important:
        `/` only changes pattern matching.
        It does NOT make command extraction case-sensitive.


===============================================================================
7. HIGH-LEVEL DATA FLOW
===============================================================================

The runtime flow is:

    argparse
        ->
    query parsing / validation
        ->
    file expansion
        ->
    command extraction
        ->
    filtering
        ->
    rendering

MORE CONCRETELY:
    1. argparse parses `-q`, `-Q`, `-A`, `-r`, `--no-color`, `--no-banner`,
       `-n`, `-o`, and file patterns
    2. `collect_ordered_query_specs()` rebuilds queries in original CLI order
    3. file globs expand through `expand_file_patterns()`
    4. each query is executed by `process_query()`
    5. `extract_command()` or `extract_commands()` finds the prompt-delimited
       command output
    6. grep / block helpers shape the output
    7. main() prints bannered output, bannerless normal output, raw output, or
       notes mode output

MULTI-QUERY ORDERING:
    Mixed `-q` and `-Q` queries are executed in the order typed on the CLI.

AND MODE:
    `-A` is applied after per-query output has been rendered for a file.
    If any query returns empty content, that file is suppressed.


===============================================================================
8. FUNCTION MAP
===============================================================================

print_banner(filename)
    Prints the yellow per-file banner used in normal mode.

extract_commands(file_path, command)
    Core extractor for all prompt-delimited matches of a command prefix.
    Returns a list of CommandMatch objects.
    This is the engine behind `+`.

extract_command(file_path, command)
    Thin wrapper over `extract_commands()` that returns only the first match.

get_regex_flags(modifiers)
    Chooses regex flags for filtering.
    Default is case-insensitive.
    `/` removes `re.IGNORECASE`.

grep_output(output, grep_pattern, regex_flags)
    Standard line-based grep behavior.
    Returns whole matching lines only.

grep_output_matches_only(output, grep_pattern, regex_flags)
    `%` helper for line mode.
    Returns only the first matched substring per matching line.

count_non_empty_lines(output)
    `#` helper.
    Returns the number of non-empty rendered lines, or empty string if no
    content remains after filtering.

_trim_match(line, grep_pattern, regex_flags)
    Internal helper used when block-mode matched lines must be trimmed.

grep_output_with_blocks(output, grep_pattern, regex_flags)
    Classic block-aware filter.
    Returns full parent/child blocks containing a match.

grep_output_with_blocks_selective(...)
    Richer block helper for `@` and `%` in block mode.
    Handles:
        - full blocks
        - parent + matched child lines only
        - trimming matching lines to matched substring only

expand_file_patterns(patterns)
    Expands globs and direct paths, preserves order, and de-duplicates.

parse_query(query_string)
    Splits a query into `command` and `pattern` on the first colon only.

parse_q_query(query_string)
    Validates and parses `-q` queries, including symbolic modifiers.

parse_query_block(query_string)
    Validates and parses `-Q` queries, including `@`.

collect_ordered_query_specs(argv)
    Reconstructs the mixed `-q` / `-Q` list in CLI order.

process_query(file_path, query_spec)
    Runs one parsed query against one file.
    It is the main execution brain:
        extract
        filter
        join multi-command results

get_query_results(file_path, all_queries)
    Runs every query for one file and preserves order.

get_file_output(file_path, args, all_queries)
    Builds the final combined output string for one file.
    Notes mode depends on this exact rendered output for deduplication.

print_unique_banner(index, total, match_count)
    Prints the notes-mode banner:
        <<< Unique Output 1 of 3 | matched 2 times >>>

build_notes_report(notes_collected)
    Builds the final NOTES REPORT block as text, including the compact
    `Command:` summary line used for provenance.

build_notes_command_summary(argv)
    Reconstructs the reusable showparse syntax for the notes report.
    Keeps query/output-shaping flags, omits `-n`, `-o`, and file targets.

validate_output_file_path(output_file_path)
    Preflight validation for `-n -o`.
    This exists specifically so bad paths fail before the user spends time
    entering notes.

main()
    argparse, validation, file expansion, notes mode, raw mode, and normal mode
    live here.

9. INTENTIONAL BEHAVIORS THAT SHOULD NOT BE "CLEANED UP"
===============================================================================

This section is here because these behaviors might look odd at a glance, but
they are intentional and user-driven.

1. `show run` PREFIX-MATCHES MULTIPLE REAL COMMANDS
    This is not a bug.
    Without `+`, the first prompt-delimited match wins.
    With `+`, all prompt-delimited matches are processed in file order.

2. INDENTED COMMAND TEXT SHOULD NOT COUNT AS A NEW COMMAND
    The parser is supposed to ignore command-looking lines inside `show history`
    and similar output.

3. `@` IS `-Q` ONLY
    This is by design because `@` only makes sense when a parent/child block
    model exists.

4. `/` ONLY CHANGES FILTER MATCHING
    It should not change how prompt-delimited command extraction works.

5. `-n` DEDUPES ON FINAL RENDERED OUTPUT
    Notes mode groups files by the actual final rendered string from
    `get_file_output()`, not by intermediate command output.
    This is why output-shaping modifiers matter so much for good notes grouping.

6. NOTES MODE CTRL+C IS QUIET
    Ctrl+C during `Notes:` should:
        - avoid a traceback
        - print a newline so the shell prompt is clean
        - avoid printing a notes report
        - exit quietly

6b. NORMAL AND RAW OUTPUT CTRL+C ARE QUIET TOO
    Ctrl+C during non-notes output rendering should:
        - avoid a traceback
        - avoid explanatory error text
        - print a newline so the shell prompt is clean
        - exit with status 130

6a. NOTES MODE SHOWS PHASE-1 PROGRESS ON STDERR
    During the pre-review file-processing pass, notes mode prints a live
    `(completed/total)` counter to stderr only.

    Why stderr:
        It keeps stdout clean for:
            - unique output banners
            - rendered notes-review content
            - `Notes:` prompts
            - the final notes report

7. `-o` IS NOTES-ONLY AND PREFLIGHT-VALIDATED
    A bad `-o` path should fail before notes entry begins.
    This protects the user from losing time on a typo.

8. NOTES REPORTS INCLUDE A COMPACT `Command:` LINE
    The final NOTES REPORT block includes a compact showparse syntax summary.
    It keeps query/output-shaping flags that affect the rendered content, but
    intentionally omits `-n`, `-o`, and file targets so the report stays
    focused on the parsing recipe.

9. RAW MODE PREFIXES NON-EMPTY LINES ONLY
    Empty lines are skipped in raw mode.

10. BANNERED MODE PRINTS QUERY RESULTS IN CLI ORDER
    Multiple queries are separated by a dashed divider.

11. `--no-color` ONLY CHANGES NORMAL-MODE BANNER STYLING
    It keeps the banner text but removes the yellow ANSI styling.
    It does not affect raw mode or notes-mode unique-output banners.

12. `--no-banner` ONLY CHANGES NORMAL-MODE FILE HEADERS
    It does not affect raw mode or notes-mode unique-output banners.
    Multiple files are still separated by a blank line.

12. SAFE CHANGE CHECKLIST
===============================================================================

IF ADDING A NEW MODIFIER:
    - keep it symbolic, not letter-based
    - update the allowed modifier sets
    - update query validation
    - decide whether it runs before or after final rendering
    - update help output in main()
    - update this guide
    - update REGRESSION_CHECKLIST.md
    - add targeted core tests and CLI tests

IF CHANGING COMMAND EXTRACTION:
    - preserve column-0 prompt matching unless intentionally redesigning it
    - preserve same-prompt output boundary behavior unless intentionally redesigning it
    - preserve zero-or-one-space prompt support unless intentionally redesigning it
    - verify `show history` / indented command text still does not create false positives
    - verify NX-OS spacing still works

IF CHANGING BLOCK LOGIC:
    - verify full-block mode still works
    - verify `@` still behaves as:
        parent matched -> full block
        child matched -> parent + matched children only
    - verify `%` trimming in block mode still only trims matched lines

IF CHANGING NOTES MODE:
    - preserve grouping by final rendered output unless intentionally redesigning it
    - preserve header format unless intentionally changing it
    - preserve compact `Command:` provenance behavior unless intentionally redesigning it
    - preserve quiet Ctrl+C handling unless intentionally redesigning it
    - preserve `-o` preflight validation unless intentionally redesigning it

IF CHANGING FIXTURES:
    - keep the IOS demo story strong
    - keep the NX-OS spacing fixture intact unless intentionally redesigning extraction support
    - re-check exact-output tests and notes grouping
    - re-check demo commands in the checklist

IF CHANGING LARGE PRACTICE FIXTURES:
    - treat `generate_large_fixtures.py` as the source of truth, not manual edits
    - keep large fixtures in `test1/data_large`, separate from `test1/data`
    - avoid synthetic prompt lines that would accidentally change command boundaries
    - preserve the original small fixture body inside the generated large file

IF CHANGING CLI HELP:
    - keep the `Modifiers:` section
    - keep examples brief but representative
    - make sure removed legacy flags do not reappear
